# KAO — Operator Onboarding

## Purpose

This document introduces a Linux operator to Kao.

The goal is to help the operator:

- understand what Kao is
- read the system scene
- act safely
- activate and deactivate runtime actors
- stabilize a confusing situation

This is not marketing material.

This is a first operational guide.

---

## First contact with Kao

Kao starts in a terminal.

The login screen displays a system scene.

This scene indicates:

- the machine in use
- the active Kao system
- the authority currently in control
- the runtime state

Before any action, the operator should read this scene.

This makes it possible to know:

- whether the system is stable
- who can act
- what a command will affect

The first step with Kao is not to execute a command.

It is to understand the present state.

---

## Read the system scene

The scene displayed by Kao must be interpreted.

Each piece of information has an operational purpose.

The operator identifies:

- machine name
- system role
- active authority
- runtime actor availability

This reading makes it possible to decide:

- whether it is safe to act
- whether a source should be inspected first
- whether a situation must be stabilized first

A poor reading of the scene can cause:

- incorrect activation
- authority conflict
- runtime drift

Understanding the system scene is a core Kao skill.

---

## First safe actions

After reading the scene, the operator can act.

The first actions should remain low risk.

It is recommended to:

- inspect available sources
- verify the active actor
- observe runtime state
- consult system information

These actions help:

- become familiar with Kao
- understand possible transitions
- avoid involuntary modification

A good practice is to prefer read commands before transition commands.

---

## Activate a runtime actor

Activating an actor changes the system state.

Before activation, the operator should:

- verify that the source is valid
- confirm the currently active authority
- understand the expected effect on runtime

A successful activation creates:

- a visible transition
- an updated system scene
- a new action capacity

After activation, it is recommended to:

- read the scene again
- confirm the active state
- avoid immediate additional transitions

Activation should remain a conscious decision.

---

## Deactivate and stabilize

Deactivating an actor helps regain control of runtime.

This action is useful when:

- an activation happened too early
- a state becomes difficult to read
- several transitions follow each other

Deactivation reduces active system complexity.

This makes it possible to:

- return to a readable situation
- prepare a safer next action
- maintain a clear hierarchy

Stabilizing does not mean stopping operation.

It means returning to a readable base before continuing.

---

## Common operator errors

The first difficulties with Kao usually come from moving too fast.

Frequent errors include:

- acting without reading the system scene
- activating an actor without source inspection
- chaining transitions without verification
- ignoring runtime instability signals

These situations can cause:

- authority confusion
- hard-to-read state
- loss of operator control

The usual correction is to:

- reduce active runtime
- read the scene again
- resume with a more methodical progression

---

## Kao operator discipline

Operating Kao requires a simple and constant method.

An effective discipline is to:

- read system state before acting
- limit simultaneous transitions
- verify the effect of each command
- maintain a clear runtime hierarchy

This approach makes it possible to:

- understand a situation quickly
- avoid unnecessary drift
- keep the system governable

With practice, this discipline becomes natural.

The terminal then becomes a stable and controlled work space.
