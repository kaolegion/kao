# KAO DOCUMENT EVOLUTION PROTOCOL

## Purpose

This document defines the canonical evolution protocol for Kao documentation.

It exists to ensure that Kao documentation remains:

- structured
- atomic
- canonical
- testable
- source-aligned
- reproducible
- clonable

This protocol is the base discipline for progressive documentation growth.

---

## Sovereign language rule

The sovereign source language of canonical Kao documentation is:

- English

This means:

- canonical doctrine is written in English first
- translations are secondary layers
- translations must follow the canonical source version
- conceptual truth is maintained in the English source

This rule exists to preserve:

- conceptual stability
- repository coherence
- contribution clarity
- reproducible evolution cycles

---

## Canonical cycle model

Kao documentation evolves through fixed cycles.

A standard cycle is:

- Day 0: canonical seed or cycle baseline
- Day 1 to Day 13: daily micro-evolution
- Day 14: consolidation day
- Day 15: new cycle baseline begins

The objective is not random growth.

The objective is controlled cognitive maturation.

---

## Daily micro-evolution rule

Each daily documentation evolution should remain:

- narrow
- explicit
- inspectable
- testable
- reversible

A daily evolution may include:

- one clarification
- one new section
- one structural correction
- one terminology alignment
- one new link between canonical concepts

A daily evolution should avoid:

- uncontrolled doctrinal branching
- vague accumulation
- multiple conceptual shifts at once
- hidden rewrites without stated intent

---

## Consolidation day rule

Day 14 is reserved for convergence.

Its role is to:

- review the previous 13 days
- merge useful refinements
- remove doctrinal duplication
- clarify naming
- stabilize structure
- prepare the next cycle seed

Consolidation is not only compression.

It is cognitive refinement.

---

## Canonical document requirements

A canonical Kao document should aim to provide:

- a clear purpose
- explicit scope
- stable terminology
- readable section hierarchy
- traceable links with related documents

A canonical document should remain:

- useful to the operator
- useful to the system
- useful to future agents
- usable in Git history
- inspectable in terminal workflows

---

## Required evolution metadata

When a canonical document changes, the evolution should be explainable through:

- change intent
- affected layer
- structural impact
- validation result

This metadata may appear in:

- sprint notes
- commit messages
- dedicated evolution logs
- consolidation summaries

---

## Validation discipline

Documentation evolution should not be treated as decoration.

It must be validated.

Validation may include:

- terminology inspection
- architecture alignment
- cross-document coherence checks
- command surface alignment
- roadmap continuity checks

When relevant, documentation changes should be checked against:

- runtime reality
- CLI surfaces
- tests
- system vocabulary
- existing doctrine

---

## Source and runtime distinction

Canonical documentation belongs to versioned source.

Generated traces, temporary notes, runtime logs, and local artifacts do not.

This distinction must remain explicit.

Documentation evolution must strengthen the source layer without polluting it with runtime noise.

---

## Long-term role

This protocol exists to help Kao evolve as a disciplined cognitive system.

The goal is not only to write documents.

The goal is to make Kao capable of:

- learning from structure
- stabilizing its knowledge
- improving through cycles
- preserving identity while evolving
