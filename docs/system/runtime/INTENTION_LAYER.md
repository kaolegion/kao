# KAO INTENTION LAYER

## Purpose

Defines the canonical role of intention inside Kao runtime architecture.

Intention is the explicit cognitive commit connecting thinking to action.

It is not:

- a raw signal
- a simple perception
- an already executed action
- a persistent memory by default

It is:

- an explicit action direction
- a runtime cognitive commitment
- a pre-routing executable formulation
- a stabilizing layer between thinking and routing

## Canonical chain

Kao runtime cognition should be understood as:

signal
→ perception
→ cognition
→ thinking
→ intention
→ routing
→ dispatch
→ execution
→ observation
→ memory

Intention exists between thinking and routing.

Thinking explores.
Intention commits.

## Role

The intention layer may:

- express what Kao is preparing to do
- reduce ambiguity before routing
- provide a stable object to router
- prevent premature or unclear execution
- carry confidence, scope and risk before action

## Runtime properties

Intention must remain:

- explicit
- compact
- traceable
- bounded
- observable
- append-only in trace form
- usable without heavy external intelligence

Intention must not become a verbose reasoning surface.

## Intention states

An intention sequence may move through the following states:

- emerging
- framed
- ready
- routed
- cancelled
- expired

## Minimal intention dimensions

A canonical intention may carry compact dimensions such as:

- target
- purpose
- scope
- confidence
- risk
- urgency

## Canonical trace

An intention trace may be represented using the canonical line:

ts|source|domain|event|payload

Examples:

1710713200|kao|intention|emerging|inspect runtime signal path
1710713201|kao|intention|framed|read-only runtime inspection
1710713202|kao|intention|ready|route local inspection
1710713203|kao|intention|routed|local shell
1710713204|kao|intention|cancelled|insufficient authority

## Relation with thinking

Thinking may compare and branch.

Intention formulates and stabilizes.

A thought should not be routed while still unstable.

## Relation with router

Router receives a prepared intention and decides execution path.

Router may consider:

- confidence
- risk
- urgency
- runtime availability

Canonical order:

thinking
→ intention
→ router
→ dispatch

## Relation with operator

The operator may:

- produce an explicit intention
- influence an implicit intention
- invalidate a current intention
- reinforce its priority

## Relation with brain

Brain may enrich intention through memory and context.

But intention must remain possible even without brain support.

## Relation with autocorrection

The intention layer helps detect:

- vague direction
- unsafe scope
- contradictory goals
- obsolete purpose

Autocorrection may reformulate or cancel before execution.

## Strategic value

Formal intention enables:

- safer routing
- stronger explicability
- bounded autonomy
- cleaner operator collaboration
- better pre-execution correction

Cognition frames.
Thinking explores.
Intention commits.
Router chooses.
Dispatcher applies.
Runtime executes.
Observation confirms.
Memory preserves.
