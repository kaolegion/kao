# KAO — Support

## Intent

This document explains how to ask for help
and how to structure useful support requests.

---

## Good support requests

A good Kao support request usually includes:

- what you were trying to do
- which file or command was involved
- what you expected
- what happened instead
- exact terminal output when relevant
- current repo path or runtime context if useful

---

## Preferred support style

Kao favors support requests that are:

- explicit
- step-by-step
- reproducible
- grounded in observed behavior

This makes diagnosis faster and safer.

---

## Useful material to include

Depending on the issue, it may help to include:

- `pwd`
- relevant `sed -n` file inspections
- exact command used
- verification output
- current `git status`
- current runtime state description

---

## Documentation first

Before opening a support request, check:

- `README.md`
- `docs/README.md`
- `CONTRIBUTING.md`
- `ROADMAP.md`
- `CODE_OF_CONDUCT.md`

If the issue is behavioral, also inspect
the current target files before assuming a previous state.

---

## Project culture reminder

Support in Kao is not only about fixing errors.

It is also about preserving:

- clarity
- convergence
- deterministic reasoning
- readable operator workflows
